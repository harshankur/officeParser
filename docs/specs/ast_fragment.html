officeParser AST Structure Specification

The officeParser library produces a structured Abstract Syntax Tree (AST) that represents the parsed document in a format-agnostic way. This tree is designed for high-fidelity reconstruction and programmatic analysis.

OfficeParserAST (Root)

Property Type Description
type SupportedFileType The type of the parsed file (docx, xlsx, pptx, pdf, rtf, odt, ods, odp, csv, md, html, epub).
metadata OfficeMetadata Global document metadata (author, title, creation/modification dates).
content OfficeContentNode[] The hierarchical tree of document content nodes.
attachments OfficeAttachment[] Unified array of binary resources (images, charts, objects) referenced in the content.
auxiliary OfficeAuxiliaryContent | undefined Out-of-band layout / template elements that are not part of the main content flow. Contains headers, footers (from DOCX), and slideMasters (from PPTX). Only populated when the respective ignoreHeadersAndFooters / ignoreSlideMasters flags are false.
warnings OfficeIssue[] Non-fatal issues encountered during the parsing phase (e.g. OCR failures, skipped nodes). Always check this after parsing if you use onWarning.
toText() function Deprecated: Synchronous utility to flatten the AST into plain text. Use the async .to('text') instead: it is the maintained renderer, and this one is its unconfigurable predecessor. At its defaults .to('text') emits everything this method emits, verified across every bundled fixture in all 12 formats in both layout modes, and it renders merged table cells correctly (this method glues them into OneThree rather than One Three). The rest is configuration, not capability: notes and image placeholders are emitted by default and switchable via textConfig.renderNotes and includeImages, whereas this method emits neither and offers no way to request them. For the closest match to this output, pass { includeImages: false, textConfig: { preserveLayout: false, renderNotes: false } }.
to(destination, config?) Promise<ConversionResult> Recommended: Universal entry point for generation. Converts the AST to text, md, html, csv, rtf, pdf, epub, or chunks. Returns a ConversionResult with { value, messages }.

Node Interpretation Guide

Every node in the content tree follows a consistent recursive structure. Children are stored in the children array, and visuals are defined in formatting.

[!NOTE] Discriminated Union: OfficeContentNode is now a discriminated union type. Narrowing on node.type (e.g., if (node.type === 'heading')) gives precise metadata inference — no type-casting needed.

OfficeContentNode

Property Type Description
type OfficeContentNodeType The functional role of the node (e.g., paragraph, table, list). Discriminant for type narrowing.
text string Flattened text content of this node and all its descendants. Useful for quick search/indexing.
children OfficeContentNode[] Recursive child nodes. Use these for high-fidelity rendering.
comments OfficeContentNode[] | undefined Comments/annotations attached to this node. Each entry is a comment node with CommentMetadata (author, date, commentId). Populated only when ignoreComments: false (default).
notes OfficeContentNode[] | undefined Footnotes, endnotes, or slide speaker notes attached to this node. Each entry is a note node. Populated only when ignoreNotes: false (default).
metadata ContentMetadata Type-specific configuration. The exact type is inferred from node.type when the discriminated union is narrowed (e.g., HeadingMetadata for type === 'heading').
formatting TextFormatting Styling applied to the node (bold, italic, size, color, etc.).
rawContent string | undefined The raw XML or RTF source markup for this specific node. (Only if enabled in config).

Core Node Types

1. Lists (The list Node)

Lists are represented as a flat sequence of list nodes. To reconstruct the visual nesting (nested ul/ol), you must interpret the metadata.

  • indentation: Integer starting at 0. Represents the structural nesting level.
  • paragraphIndentation: Physical paragraph indentation formatting (e.g., left, hanging) measured in twips.
  • listType: 'ordered' or 'unordered'.
  • listId: A unique identifier for the list group. Use this to determine if adjacent items belong to the same logical list.
  • itemIndex: (Ordered lists only) The 0-based index of the item.
[!TIP] Interpretation Logic: Maintain a "List Stack". When indentation increases, push a new list container. When it decreases, pop the stack. If listId changes at the same indentation, start a new list block.

2. Tables

  • table: The container. Carries TableMetadata with optional anchorIds and align ('left' | 'center' | 'right', round-tripped from HTML data-align).
  • row: A horizontal group of cells.
  • cell: A container for other content (paragraphs, images, or even nested tables). Use metadata for rowSpan, colSpan, and backgroundColor (populated from DOCX w:shd fills; also read from HTML colspan/rowspan attributes so merged cells survive an HTML round-trip).

3. Structural Containers

These nodes represent large-scale document sections:

  • slide: Container for PPTX/ODP slides. metadata.slideNumber provides the position. Slide speaker notes are now attached via slideNode.notes[] instead of being inserted as top-level note nodes.
  • page: Container for PDF pages. metadata.pageNumber provides the position.
  • sheet: Container for XLSX/ODS spreadsheets. metadata.sheetName provides the label.

4a. Layout & Template Nodes (Auxiliary)

These nodes are not part of the main content tree. They live under ast.auxiliary:

  • header: A header element from a DOCX file. Carries HeaderFooterMetadata.type ('default', 'first', 'even'). Found in ast.auxiliary.headers[].
  • footer: A footer element from a DOCX file. Same metadata. Found in ast.auxiliary.footers[].
  • slideMaster: A PPTX slide master. Carries SlideMetadata. Found in ast.auxiliary.slideMasters[].

4b. Annotation Nodes

  • comment: An inline comment or annotation attached to a document node via node.comments[]. Carries CommentMetadata: author, initials, date, commentId.

4c. Markdown-Dialect Nodes

These node types are produced by the extended Markdown/HTML dialect support (task lists, admonitions, embeds, definition lists) and round-trip symmetrically between Markdown, HTML, and EPUB.

  • admonition: A callout/alert block (GitHub > [!NOTE] or GLFM :::note ... ::: on import). Carries AdmonitionMetadata.admonitionType ('note' | 'tip' | 'important' | 'warning' | 'caution') and an optional title. Has children for its block content.
  • embed: A rich embed such as a YouTube video. Carries EmbedMetadata: embedType, videoId, url, width, align.
  • definitionList / definitionTerm / definitionDescription: Markdown Extra-style definition lists (Term\n: Definition). The list node's children alternate term and description nodes.
  • list task-list items: ListMetadata.isTask and .checked mark a GFM task-list item (- [x] Done).
  • Text-node metadata additions: TextMetadata.citationKey (Pandoc/MultiMarkdown [@key]), .wikilink (Obsidian [[page]]), .math ('inline' | 'block' LaTeX), and .abbreviationTitle (Markdown Extra *[HTML]: ... abbreviations).

4. Rich Content

  • image: References an item in the root attachments array via metadata.attachmentName. Also carries optional width and align ('left' | 'center' | 'right'), round-tripped from HTML data-width/data-align or an inline style="width:...".
  • chart: References a chart attachment. Contains structured chartData in metadata if requested.

5. Breaks (The break Node)

Breaks are added as break nodes when includeBreakNodes is enabled in the configuration. They represent physical layout changes within the document flow.

  • breakType:
    • 'textWrapping': Standard line break (Shift+Enter). This is the default break type.
    • 'carriageReturn': Explicit hard return (w:cr), often used for legacy compatibility.
    • 'page': Hard page break, forcing subsequent content to a new page.
    • 'column': Column break, moving text to the top of the next column in multi-column layouts.
    • 'lastRenderedPage': A "soft" page break marker indicating where a page naturally ended during the last render.
  • clear: (Only for textWrapping) Specifies where the next available line should start when wrapping around floating objects (all, left, right, none).
[!NOTE] Plain Text Conversion: Even though break nodes have no text property, both ast.to('text') and the deprecated ast.toText() automatically convert them into newlines (\n) or the configured newlineDelimiter to preserve the document's visual structure.

OfficeIssue Interface

All warnings and errors from both parsing (ast.warnings) and generation (result.messages) use this unified shape:

interface OfficeIssue {
    type: 'warning' | 'info' | 'error';
    code: OfficeWarningType | OfficeErrorType; // typed enum, e.g. 'OCR_FAILED'
    message: string;
    node?: OfficeContentNode; // the node that triggered the issue, if applicable
    details?: any;            // original error object or extra context
}

Use issue.code for programmatic handling (e.g. if (issue.code === 'OCR_FAILED') { ... }).

}

OfficeMetadata Extensions

Two new fields added to the root ast.metadata object:

  • keywords (string): Keywords associated with the document, extracted from document properties.
  • nativeProperties (Record<string, any>): All format-specific metadata extracted verbatim. Examples:
    • HTML: All <meta> tag values indexed by name / property
    • DOCX: docProps/app.xml entries (e.g., Pages, Application)
    • XLSX: docProps/app.xml entries
    • PDF: Raw pdfjs document info dict and XMP metadata object
    • Markdown: YAML frontmatter key-value pairs

Programmatic Traversal Best Practices

  1. Depth-First Search: Use recursion to walk the children tree for rendering.
  2. Fallback to .text: If you only need plain text, use the text property of any node—it is pre-calculated to include all text from its children.
  3. Map Attachments First: Before rendering, create a Map from the attachments array using the name property for O(1) lookups when encountering image nodes.
  4. Traverse node.notes and node.comments: These arrays are separate from children and will not appear during a standard DFS of children. Iterate them explicitly if you need footnotes, endnotes, or comments.
  5. Access Auxiliary Content via ast.auxiliary: Headers, footers, and slide masters are not in ast.content. Access them via ast.auxiliary?.headers, ast.auxiliary?.footers, or ast.auxiliary?.slideMasters.
  6. Type Narrowing: Narrow on node.type before accessing node.metadata to benefit from precise TypeScript inference (e.g., node.type === 'cell' && node.metadata?.backgroundColor).
