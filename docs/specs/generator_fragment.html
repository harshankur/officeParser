OfficeGenerator & RAG Specification

v7.0.0 introduces a powerful document generation engine and native RAG chunking suite. This specification details how to transform ASTs into usable formats and prepare documents for AI pipelines.

OfficeConverter.convert()

The primary entry point for one-step transformations. It automatically handles parser settings (like enabling attachments if needed for the generator).

import { OfficeConverter } from 'officeparser';

const { value, messages } = await OfficeConverter.convert('input.docx', 'md', {
    generatorConfig: {
        includeFormatting: false
    }
});

OfficeGenerator.generate()

The core generation engine. Use this when you already have an AST and want to transform it into multiple formats without re-parsing.

import { OfficeGenerator } from 'officeparser';

const html = await OfficeGenerator.generate(ast, 'html');
const pdf = await OfficeGenerator.generate(ast, 'pdf');

GeneratorConfig

Unified configuration for all output formats. Common settings apply to every generator.

Property Type Default Description
includeFormatting boolean true Whether to include visual styles (colors, sizes) in the output.
generateIds boolean true Automatically generates unique slug-based IDs for headings (for anchors).
styleMap string[] | object[] (Standard) Custom mapping for document styles (e.g., 'Heading 1' -> 'h1').
onNode function undefined Hook to intercept every node.
Pro Tip: Return false to skip, a string to override, or mutate node and return void to continue.

🚀 Pro User: Advanced Node Manipulation

The onNode callback is your "Swiss Army Knife" for generation. It allows for three main workflows:

  • Filtering: Return false to prune branches of the document.
  • Overriding: Return a string to provide custom HTML/Markdown for a specific node type.
  • Interception: Mutate node.text or node.metadata and return void to use the standard generator with your modifications.
// Example: Dynamic redaction and custom callouts
onNode: (node) => {
    if (node.text?.includes('CONFIDENTIAL')) {
        node.text = node.text.replace('CONFIDENTIAL', '***');
    }
    if (node.metadata?.style === 'Alert') {
        return `<div class="alert">${node.text}</div>`;
    }
}

RAG & Chunking Strategies

When the destination is chunks, the library returns an array of OfficeChunk objects with rich metadata.

1. Document Structure Chunking (Recommended)

Leverages the AST to split at natural boundaries like paragraphs or headings. This is the most semantically stable strategy.

const { value: chunks } = await ast.to('chunks', {
    chunksConfig: {
        strategy: 'document-structure',
        splitBy: 'heading',
        maxChunkSize: 1000,
        tableSplitStrategy: 'row' // Repeats headers in every chunk!
    }
});

2. Fixed-Size Chunking

Splits text by character count with a sliding window overlap. Best for strict token-limit adherence.

Style Mapping (The Semantic Bridge)

officeParser uses a "Universal Intermediate Language" (HTML tags) to define the semantic meaning of document styles.

Legacy String DSL

["p[style-name='Heading 1'] => h1", "p[style='Quote'] => blockquote"]

Structured Style Mappings (Recommended)

[
  { 
    selector: { nodeType: 'paragraph', attributes: { style: 'Heading 1' } }, 
    output: { tag: 'h1', classes: ['main-title'] } 
  }
]

ConversionResult

All generation methods return a ConversionResult object. This ensures that non-fatal issues (like missing fonts or layout approximations) are reported without failing the entire process.

  • value: The generated content (string, Buffer, or Chunk array).
  • messages: Array of warnings, info, and errors encountered.
